LifeOS

A personal operating system that treats Markdown and YAML as the source of truth, adds an Android interface, and synchronizes confirmed events with Google Calendar through a defensive Python bridge.

Problem

Keep personal data portable without losing product ergonomics.

Conventional task managers lock structure inside a database or cloud account. LifeOS keeps human-readable files as the canonical state, while the application layer adds search, notifications, event workflows and calendar integration.

01 / PORTABILITY

Files stay useful

Projects and events remain readable and editable outside the application.

02 / SAFETY

Writes must be defensive

The bridge checks for concurrent changes and uses temporary files, fsync and atomic replacement.

03 / INTEGRATION

Calendar without duplication

Stable synchronization identifiers and conflict recovery prevent repeated event creation.

Architecture

Four layers, one explicit source of truth.

The Android client never becomes the only place where data exists. Markdown/YAML remains canonical; integrations work through validated and recoverable boundaries.

01 / CLIENTAndroid · ComposeNavigation, local notifications and product UI
02 / SOURCEMarkdown · YAMLPortable projects, events and metadata
03 / BRIDGEPythonValidation, OAuth, safe writes and synchronization
04 / EXTERNALGoogle Calendar APIOwned event scope and stable event identifiers

Engineering decisions

Failure cases were designed before polish.

AI tools helped with decomposition, implementation alternatives and review. The final behavior was constrained by specifications, tests and real-device checks.

Optimistic conflict detection

Before updating frontmatter, the bridge compares current disk content with the version that was originally loaded.

Atomic file replacement

A temporary file is flushed and synchronized before os.replace changes the original path.

Idempotent synchronization

A Google-compatible sync UID and recovery from HTTP 409 handle the case where an event already exists.

Narrow OAuth scope

The bridge requests access only to events owned by the application instead of the entire calendar account.

Selected code

Protect the user's file before changing it.

This shortened fragment comes from the real Python bridge. It combines an optimistic concurrency check with an atomic filesystem update.

def update_properties_atomic(document, updates):
    current = document.path.read_text(encoding="utf-8")
    if current != document.content:
        raise BridgeError(
            f"Document changed during operation: {document.path}"
        )

    fd, temp_name = tempfile.mkstemp(
        prefix=document.path.name + ".",
        suffix=".tmp",
        dir=document.path.parent,
    )
    with os.fdopen(fd, "w", encoding="utf-8") as stream:
        stream.write(rendered_content)
        stream.flush()
        os.fsync(stream.fileno())
    os.replace(temp_name, document.path)
